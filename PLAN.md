# Notes App Backend — Implementation Plan

## Context

Greenfield take-home: build a multi-user notes REST API and deploy it to Render.com. The user has chosen **TypeScript + Node.js + Express + PostgreSQL + Prisma ORM** and selected **pinned notes + reordering** as the required creative feature. They will maintain git history themselves, so the plan is structured in **discrete, commit-sized stages**.

The graded surface is API correctness, edge-case validation, and security — so the plan emphasizes input validation, ownership checks, and authentication boundaries over architectural sophistication. Each stage is independently testable and small enough to commit on its own.

---

## Stack & Conventions

- **Runtime:** Node.js 20+
- **Language:** TypeScript (strict mode). `tsx` for dev, `tsc` → `dist/` for prod
- **HTTP:** Express 4
- **DB:** PostgreSQL via **Prisma ORM**. Migrations via `prisma migrate`. PrismaClient singleton.
- **Auth:** `bcrypt` (cost 10) for hashing, `jsonwebtoken` HS256 with 7-day expiry
- **Validation:** `zod` — schemas double as TS types via `z.infer<typeof schema>`
- **IDs:** UUIDs (`@default(uuid())` — generated client-side by Prisma, unguessable, slug-safe)
- **Email case:** Always normalize to lowercase on insert and on query (no `CITEXT`, Prisma doesn't model it natively)
- **Errors:** Centralized error middleware returns `{ message: "..." }` + correct status code; never leak stack traces or "user not found" vs "wrong password" distinctions
- **Ownership privacy:** Non-accessible notes return **404** (not 403) — don't leak existence
- **Migrations:** Prisma migrations run automatically on deploy via `prisma migrate deploy` in the start script

## Prisma Schema (`prisma/schema.prisma`)

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id           String      @id @default(uuid()) @db.Uuid
  email        String      @unique               // normalized lowercase at app layer
  passwordHash String      @map("password_hash")
  createdAt    DateTime    @default(now()) @map("created_at")

  notes        Note[]
  sharedNotes  NoteShare[]

  @@map("users")
}

model Note {
  id        String      @id @default(uuid()) @db.Uuid
  ownerId   String      @map("owner_id") @db.Uuid
  title     String
  content   String      @default("")
  pinned    Boolean     @default(false)
  position  Int         @default(0)
  createdAt DateTime    @default(now()) @map("created_at")
  updatedAt DateTime    @updatedAt      @map("updated_at")

  owner     User        @relation(fields: [ownerId], references: [id], onDelete: Cascade)
  shares    NoteShare[]

  @@index([ownerId])
  @@index([ownerId, pinned, position, updatedAt])
  @@map("notes")
}

model NoteShare {
  noteId   String   @map("note_id") @db.Uuid
  userId   String   @map("user_id") @db.Uuid
  sharedAt DateTime @default(now()) @map("shared_at")

  note     Note     @relation(fields: [noteId], references: [id], onDelete: Cascade)
  user     User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@id([noteId, userId])
  @@index([userId])
  @@map("note_shares")
}
```

**Notes on schema choices**
- We use snake_case `@map` for column/table names so the underlying SQL stays conventional, while TypeScript fields stay camelCase.
- `@db.Uuid` makes Postgres store as native UUID type (16 bytes, not 36-char TEXT).
- No `CITEXT` — we lowercase emails in the app layer (one helper). Trade: small app-side discipline for one fewer Postgres extension.
- No `tsvector` column — Stage 6 search uses Prisma's `fullTextSearch` preview feature or falls back to `contains` (ILIKE). See Stage 6.

You already ran `migrations/001_init.sql` against `notes_app` directly. Before `prisma migrate dev`, reset the public schema so Prisma owns the truth:

**Files:** `package.json`, `tsconfig.json`, `.gitignore`, `.env.example`, `prisma/schema.prisma`, `src/config.ts`, `src/db.ts`, `src/errors.ts`, `src/index.ts` (minimal app with `/health`)

```bash

npm i -D typescript tsx @types/node @types/express @types/bcrypt @types/jsonwebtoken prisma
npx tsc --init                    # generate tsconfig.json (then edit strict: true, target: ES2022)
npx prisma init --datasource-provider postgresql
```

**`package.json` scripts:**
```json
{
  "scripts": {
    "dev": "tsx watch src/index.ts",
    "build": "prisma generate && tsc",
    "start": "prisma migrate deploy && node dist/index.js",
    "prisma:migrate": "prisma migrate dev",
    "prisma:studio": "prisma studio"
  }
}
```

**`tsconfig.json` key settings:** `"target": "ES2022"`, `"module": "commonjs"`, `"strict": true`, `"esModuleInterop": true`, `"outDir": "dist"`, `"rootDir": "src"`, `"resolveJsonModule": true`, `"skipLibCheck": true`.

**Code:**
- `db.ts`: `export const prisma = new PrismaClient()` — singleton.
- `config.ts`: reads `DATABASE_URL`, `JWT_SECRET`, `PORT` from env; validates via zod; fails fast if missing. Exports typed `config` object.
- `errors.ts`: `class AppError extends Error` with `status` + `message`; `errorHandler(err, req, res, next)` middleware that maps `AppError` → its status, `ZodError` → 400, else 500. Hide stack in prod.
- `index.ts`: bootstraps express, JSON body parser, mounts `/health`, registers error middleware **last**, calls `prisma.$connect()` then `app.listen()`.

**Verification:**
```bash

npm run dev                               # boots server
curl localhost:3000/health                # expect {"status":"ok"}
psql notes_app -c '\dt'                   # expect users, notes, note_shares
```

### Stage 2 — Auth (one commit)
**Files:** `src/middleware/auth.ts`, `src/routes/auth.ts`, `src/validation/schemas.ts`

- `schemas.ts`: `registerSchema = z.object({ email: z.string().email(), password: z.string().min(8) })`. Same for `loginSchema`. Export `type RegisterBody = z.infer<typeof registerSchema>`.
- `POST /register`:
  - `registerSchema.parse(req.body)` (throws ZodError → 400 via centralized handler)
  - Normalize email to lowercase
  - `prisma.user.findUnique({ where: { email } })` — if exists → `409 { message: "Email already registered" }`
  - `bcrypt.hash(password, 10)`, `prisma.user.create(...)`
  - Return `201 { message: "User registered successfully" }`
- `POST /login`:
  - Validate body
  - `prisma.user.findUnique({ where: { email: email.toLowerCase() } })`
  - If missing OR `!bcrypt.compare(...)` → `401 { message: "Invalid email or password" }` (same response for both — prevents enumeration)
  - Sign JWT `{ sub: user.id, email }` with 7d expiry
  - Return `200 { access_token }`
- `requireAuth(req, res, next)` middleware: parse `Authorization: Bearer <token>`, verify, attach typed `req.user = { id, email }` (extend Express `Request` via declaration merging in `src/types/express.d.ts`); on failure → `401 { message: "Unauthorized" }`

**Verification:** Register → login → token returned. Bad password → 401. Duplicate email → 409. Missing/invalid token on a protected endpoint → 401.

### Stage 3 — Notes CRUD (one commit)
**Files:** `src/routes/notes.ts` (initial — list/get/create/update/delete)

All routes under `requireAuth`.

- `POST /notes` — body `{ title, content? }`. Title 1–255 chars, content string (≤ 100k). `prisma.note.create({ data: { ownerId: req.user.id, title, content } })`. Returns `201` + full note.
- `GET /notes` — returns owned notes **AND** notes shared with the user. Use:
  ```ts
  prisma.note.findMany({
    where: { OR: [{ ownerId: req.user.id }, { shares: { some: { userId: req.user.id } } }] },
    orderBy: [{ pinned: 'desc' }, { position: 'asc' }, { updatedAt: 'desc' }],
  })
  ```
  (Pagination added in Stage 6.)
- `GET /notes/:id` — UUID validation via zod. Same OR filter. If not found → `404 { message: "Note not found" }`.
- `PUT /notes/:id` — body `{ title?, content? }`. Use `prisma.note.update` with `where: { id, ownerId: req.user.id }` — non-owner update throws `P2025` (record not found) → map to 404 in error handler.
- `DELETE /notes/:id` — same ownership guard via compound where. Returns `204`. Cascades to `note_shares` via schema.

**Edge cases:** Invalid UUID → 400 (zod). Empty body on PUT → 400. Title > 255 chars → 400. P2025 from Prisma → 404.

**Verification:** Create note as user A, fetch list, fetch by ID, update, delete. As user B, all of A's note IDs return 404.

---

### Stage 4 — Sharing (one commit)
**Files:** `src/routes/notes.ts` (add share handler)

- `POST /notes/:id/share` — body `{ share_with_email }` (zod-validated email)
  - Check note exists & user is owner: `prisma.note.findFirst({ where: { id, ownerId: req.user.id } })` → if null, `404`
  - `email = share_with_email.toLowerCase()`; if equals `req.user.email` → `400 { message: "Cannot share a note with yourself" }`
  - `prisma.user.findUnique({ where: { email } })` → if null, `404 { message: "Recipient user not found" }`
  - `prisma.noteShare.upsert({ where: { noteId_userId: { noteId, userId } }, create: {...}, update: {} })` → idempotent
  - Returns `200 { message: "Note shared with <email>" }`

**Verification:** A creates note, shares with B. B's `GET /notes/:id` succeeds. B's `PUT` returns 404 (only owner edits). A re-shares with B → still 200 (idempotent). Share with self → 400. Share with unknown email → 404.

### Stage 5 — Creative feature: pinned + reordering (one commit)
**Files:** `src/routes/notes.ts` (add pin + reorder handlers)

- `PUT /notes/:id/pin` — body `{ pinned: boolean }`. Owner-only. `prisma.note.update({ where: { id, ownerId }, data: { pinned } })`. Returns `200` + updated note.
- `PUT /notes/reorder` — body `{ note_ids: string[] }`. Run in a transaction:
  ```ts
  await prisma.$transaction(
    note_ids.map((id, idx) =>
      prisma.note.updateMany({
        where: { id, ownerId: req.user.id },   // ownership guard inside the update
        data: { position: idx },
      })
    )
  );
  ```
  `updateMany` returns `{ count: 0 }` for non-owned IDs (silently skipped — UI list may include shared notes). Total updated count is the sum.
- `GET /notes` already orders by `pinned DESC, position ASC, updatedAt DESC`.

**Why this feature:** Most note-taking products live or die on quick access to important notes. Pinning + custom ordering is small surface area but high user value, and exercises ordered-update logic (transaction + multi-row UPDATE with ownership guard) beyond basic CRUD.

**Verification:** Pin two notes → they appear first in `GET /notes`. Reorder via API → order in GET reflects new positions. Try to reorder a non-owned note ID — silently ignored, owned notes still reorder correctly.

### Stage 6 — Stretch goals: pagination + search + Docker (one commit, or two)
**Files:** `src/routes/notes.ts` (pagination), `src/routes/search.ts`, `Dockerfile`, `render.yaml`, `.dockerignore`

- **Pagination on GET /notes:** Query params `?limit=20&offset=0` (defaults: limit 20, max 100). Response stays a plain array (per spec); pagination metadata via `X-Total-Count` header. Use `prisma.note.count` + `findMany({ skip, take })` in a `$transaction` so the count is consistent with the page.
- **Search:** `GET /search?q=keyword`. Two implementation choices:
  - **Simple (recommended for take-home):** Use `contains` with `mode: 'insensitive'` on title and content:
    ```ts
    prisma.note.findMany({
      where: {
        AND: [
          { OR: [{ ownerId }, { shares: { some: { userId: ownerId } } }] },
          { OR: [{ title: { contains: q, mode: 'insensitive' } }, { content: { contains: q, mode: 'insensitive' } }] },
        ],
      },
    })
    ```
    Pros: zero extra config, works immediately. Cons: no ranking, no stemming.
  - **Better (optional):** Enable Prisma's `previewFeatures = ["fullTextSearchPostgres"]`, use `search: q` on a `@@fulltext` index. Adds GIN index via Prisma. Trade: bumps deploy complexity slightly.
- **Docker:** Multi-stage Dockerfile — `node:20-alpine`, install deps + copy source + `npm run build` in builder stage, copy `dist/` + `prisma/` + `node_modules` (production only) to runner. Run `prisma migrate deploy` on container start.
- **render.yaml:** Defines a Node web service + Postgres add-on. `JWT_SECRET` as generated secret. `DATABASE_URL` wired from the Postgres add-on. `buildCommand: npm install && npm run build`. `startCommand: npm start` (which already runs `prisma migrate deploy && node dist/index.js`).

**Verification:** Create 25 notes; `GET /notes?limit=10` returns 10 with `X-Total-Count: 25`. Search for a word from one note returns that note. `docker build .` succeeds locally; container runs against local Postgres.

## End-to-End Verification

After all stages, this sequence should pass against the deployed Render URL:

1. `POST /register` (user A) → 201
2. `POST /register` (user B) → 201
3. `POST /login` (A) → 200, capture token
4. `POST /notes` (A, token) → 201, capture note ID
5. `GET /notes` (A) → 200, list contains the note
6. `GET /notes/{id}` (A) → 200
7. `PUT /notes/{id}` (A) → 200, updated content
8. `POST /notes/{id}/share` with B's email (A) → 200
9. `GET /notes/{id}` (B, token) → 200
10. `PUT /notes/{id}` (B, token) → 404 (only owner edits)
11. `PUT /notes/{id}/pin {pinned:true}` (A) → 200, GET /notes lists pinned first
12. `GET /search?q=<word>` (A) → 200, returns the matching note
13. `DELETE /notes/{id}` (A) → 204; B's `GET /notes/{id}` → 404
14. `GET /about` → 200 with required JSON shape
15. `GET /openapi.json` → 200 with valid OpenAPI 3.0 doc

`tests/smoke.ts` (Stage 7) automates this.
